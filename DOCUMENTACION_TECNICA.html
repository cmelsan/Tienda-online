Sección 01

Diagrama Entidad-Relación

La base de datos de ÉCLAT Beauty está implementada en PostgreSQL (Supabase) y consta de 18 tablas con claves foráneas, restricciones CHECK y Row Level Security (RLS) activado en todas ellas. A continuación se representan las entidades agrupadas por dominio.

1.2 Tabla de relaciones

Todas las claves foráneas tienen restricción ON DELETE implícita de Supabase. Las relaciones con user_id nullable soportan pedidos de clientes guest.

Sección 02

Justificación del Stack Tecnológico

Cada tecnología fue seleccionada por criterios concretos de rendimiento, escalabilidad, coste y adecuación a e-commerce. A continuación se justifica cada elección frente a sus alternativas.

Tecnología Por qué se eligió
Astro 5 v5.0.3 · Framework web Astro permite un modelo híbrido SSG + SSR en el mismo proyecto:
  • Páginas estáticas (FAQ, Contacto, Sobre Nosotros) se generan una sola vez en build → carga instantánea + SEO óptimo
  • Catálogo, carrito y admin se renderizan en servidor → datos siempre frescos (stock, precios, flash sales)
  • Arquitectura de islas: sólo los componentes interactivos cargan JavaScript; el resto es HTML puro
Alternativa descartada: Next.js (React-only, overhead mayor; Astro se integra con cualquier framework de UI).
React 18 v18.3.1 · UI interactiva Usado exclusivamente para componentes que requieren interactividad: carrito, checkout multi-paso, filtros del catálogo, formularios admin. React 18 aporta Concurrent Mode y useTransition para actualizaciones de UI no bloqueantes.
Nanostores Estado global Gestión del carrito persistente entre navegaciones (sin perder el carrito al cambiar de página en SSR).
  • Framework-agnostic: funciona igual en componentes Astro y React
  • @nanostores/persistent: sincroniza automáticamente con localStorage
  • Solo ~1 KB frente a los 8 KB de Redux o 3 KB de Zustand
Alternativa descartada: Redux (sobre-ingeniería para este caso de uso).
Supabase PostgreSQL + Auth + RLS Proporciona un backend completo sin servidor propio:
  • PostgreSQL real con soporte a JSONB, arrays, constraints y triggers
  • Row Level Security (RLS): los datos están protegidos a nivel de BD, no sólo en la API
  • RPCs con SECURITY DEFINER: operaciones atómicas (crear pedido, reducir stock) sin exponer la service_role key
  • Auth integrada: JWT + refresh tokens, sin implementación propia
Alternativa descartada: Firebase (NoSQL → consultas relacionales complejas como el dashboard son ineficientes).
Stripe Pagos + Webhooks Estándar del sector para pagos en línea:
  • Checkout Sessions: el formulario de pago es gestionado por Stripe (PCI-DSS compliant, nunca pasan datos de tarjeta por nuestros servidores)
  • Webhook checkout.session.completed: confirma el pago de forma asíncrona y fiable, desencadenando la creación de pedido + factura
  • Refunds API: reembolsos totales/parciales; sin amount → devuelve exactamente lo cobrado (evita errores con cupones)
Alternativa descartada: PayPal (API menos consistente, UX inferior en Europa).
Brevo (Sendinblue) Email transaccional Envío de emails transaccionales con adjuntos PDF:
  • Confirmación de pedido (con factura adjunta), notificación de envío, cancelación, nota de abono
  • API sencilla que acepta adjuntos binarios (Buffer de Node.js) → ideal para PDFs de PDFKit
  • Plan gratuito generoso: 300 emails/día
Alternativa descartada: SendGrid (más complejo de configurar para adjuntos binarios).
Cloudinary Storage de imágenes Gestión optimizada de imágenes de producto:
  • Transformaciones on-the-fly: redimensionado, conversión a WebP, compresión
  • CDN global: entrega rápida sin importar la ubicación del usuario
  • Upload firmado: el admin sube directamente desde el navegador con una firma generada en servidor → sin pasar las imágenes por nuestro servidor
Alternativa descartada: Supabase Storage (sin CDN ni transformaciones automatizadas).
PDFKit Generación de PDF Generación de facturas y notas de abono directamente en Node.js:
  • Sin dependencias de navegador (funciona en API routes de Astro SSR)
  • Layout programático con posicionamiento exacto, fuentes y colores de marca
  • Output en Buffer → adjuntable directamente en el email de Brevo
Tailwind CSS 3 v3.4.17 · Estilos Desarrollo rápido de UI con diseño consistente:
  • Clases utilitarias → sin CSS en ficheros separados, menor riesgo de conflictos
  • PurgeCSS incluido: bundle de CSS final mínimo (~8 KB)
  • Responsive de forma declarativa con prefijos sm:, md:, lg:
Recharts Dashboard KPIs Gráficas para el panel de administración (ventas por mes, pedidos por estado, top productos). Basado en D3 pero con API de componentes React declarativa. Alternativa descartada: Chart.js (menos integrable con React).
Coolify + Docker Despliegue en VPS Plataforma self-hosted similar a Heroku/Railway pero en servidor propio:
  • Imagen Docker multi-stage con Node standalone (sin overhead de Next.js server)
  • SSL automático con Let's Encrypt, proxy Caddy
  • Redeploys automáticos desde GitHub al hacer push a main
  • Coste fijo del VPS vs. coste variable de Vercel/Railway a escala
Sección 03

Flujo de Facturación

El sistema de facturación de ÉCLAT Beauty cubre tres escenarios principales: compra exitosa, cancelación y devolución parcial post-entrega. Todos los precios se almacenan en céntimos de euro (enteros) para evitar errores de coma flotante.

Nota sobre cupones y reembolsos:

Cuando se aplica un cupón, Stripe cobra el importe ya descontado. Los reembolsos se realizan siempre sin pasar el parámetro amount (reembolso total) o con el importe exacto introducido por el admin (parcial). Esto evita el error "refund amount exceeds charge amount" que ocurriría si se intentara reembolsar el total bruto del pedido.

3.1 Flujo de Compra — Happy Path
1
Cliente Añade productos al carrito

El carrito se almacena en Nanostores (persistido en localStorage) y se sincroniza con la tabla carts de Supabase. Si el cliente tiene sesión, el carrito queda vinculado a su user_id; si no, se usa un session_id anónimo. Al hacer login, ambos carritos se fusionan.

2
Cliente Introduce dirección y aplica cupón (opcional)

El cupón se valida en tiempo real contra la tabla coupons: se comprueba que esté activo, dentro del rango de fechas, que no haya superado max_uses y que el importe del carrito supere min_purchase_amount. El descuento se muestra antes de confirmar el pago.

3
Sistema (API) Crea Checkout Session en Stripe

El endpoint POST /api/create-checkout-session genera una sesión de Stripe con el importe ya descontado por el cupón. Stripe gestiona el formulario de pago de forma segura (PCI-DSS compliant). Al completar el pago, Stripe redirige al cliente a /checkout/success.

WH
Stripe → Webhook Evento checkout.session.completed

Stripe envía una llamada POST a /api/webhooks/stripe. El servidor verifica la firma HMAC con STRIPE_WEBHOOK_SECRET para garantizar la autenticidad. Este mecanismo asíncrono es más fiable que el redirect del cliente (que podría cerrarse antes de completar).

4
Sistema (RPC) Creación atómica del pedido (create_order_from_session_atomic)

Un RPC de Supabase con SECURITY DEFINER ejecuta en una única transacción:

  • Inserta fila en orders con status = 'paid'
  • Inserta todas las filas en order_items
  • Reduce el stock de cada producto de forma atómica (sin race conditions)
  • Registra el uso del cupón en coupon_usage e incrementa current_uses
  • Vacía el carrito en carts

Si alguna operación falla, la transacción completa se revierte (ROLLBACK).

5
Sistema Generación de factura PDF y envío de email

Tras confirmar la creación del pedido:

  • PDFKit genera la factura con número correlativo FAC-YYYY-NNNN, líneas de producto, IVA 21%, descuento del cupón y total
  • La factura se inserta en la tabla invoices (type = 'invoice')
  • Brevo envía el email de confirmación con la factura PDF adjunta como Buffer
3.2 Flujo de Cancelación con Reembolso
1
Admin / Cliente Solicita cancelación del pedido

Sólo se pueden cancelar pedidos con status = 'paid'. Los pedidos ya enviados (shipped) o entregados (delivered) no pueden cancelarse desde esta vía; deben gestionarse como devolución.

2
Stripe Reembolso total automático

Se llama a stripe.refunds.create({ payment_intent: id }) sin pasar el parámetro amount. Stripe reembolsa exactamente lo que el cliente pagó, incluyendo el descuento del cupón. Esto evita el error de intentar reembolsar más de lo cobrado.

3
Sistema (RPC) Actualización atómica en BD (admin_cancel_order_atomic)

En una única transacción:

  • orders.status → 'cancelled'
  • Restaura el stock de todos los order_items
  • Registra el cambio en order_status_history
4
Sistema Nota de abono PDF y email de cancelación

PDFKit genera una nota de abono (type = 'credit_note', número ABN-YYYY-NNNN) que referencia la factura original mediante reference_invoice_id. Brevo envía el email de cancelación con la nota de abono adjunta.

3.3 Flujo de Devolución Parcial Post-Entrega
1
Cliente Solicita devolución desde "Mi Cuenta"

El cliente selecciona uno o varios ítems del pedido entregado y proporciona el motivo. Cada ítem queda con return_status = 'requested' en la tabla order_items. El pedido pasa a status = 'return_requested'.

2
Admin Revisa y aprueba/rechaza desde /admin/devoluciones

El admin puede aceptar cada ítem (cambia a 'approved') o rechazarlo ('rejected'). También puede introducir un importe de reembolso personalizado para devoluciones parciales.

3
Stripe Reembolso parcial o total

Si es devolución total: stripe.refunds.create({ payment_intent }) sin amount.

Si es parcial: stripe.refunds.create({ payment_intent, amount: importeEnCentimos }). Stripe valida que el importe no supere lo cobrado originalmente.

4
Sistema (RPC) Procesamiento atómico (admin_process_return)

El RPC filtra exclusivamente los ítems con return_status IN ('requested', 'approved') para evitar restaurar stock de ítems rechazados o ya procesados:

  • Restaura el stock sólo de los ítems aprobados
  • Marca esos ítems como return_status = 'refunded'
  • Si todos los ítems son refunded → pedido 'refunded'
  • Si quedan ítems sin devolver → pedido 'partially_refunded'
  • Registra el reembolso en refunds_log
5
Sistema Nota de abono y email al cliente

PDFKit genera una nota de abono parcial (credit_note_scope = 'partial') con los ítems devueltos y el importe reembolsado. Se inserta en invoices y se envía por email a través de Brevo.
